# 架构说明

## 1. 目标定位

PolicyAnalyzerPro 的目标不是做通用聊天工具，而是做一套能在政务、研究、内网环境中稳定运行的离线桌面分析器。当前设计优先保证：

- 不联网
- 不拖死桌面
- 可被 PyInstaller 打包
- 面向长文本分析与版本演变比对
- 可以直接导入常见文档并导出正式报告
- 可以在首次部署时主动发现缺失环境而不是运行时才报错

## 2. 模块职责

### `main.py`
- 负责应用入口。
- 在 `QApplication` 初始化前开启高分屏适配。
- 设置全局异常钩子，避免崩溃闪退无提示。
- 调用 `multiprocessing.freeze_support()` 兼容 Windows 打包环境。
- 在主窗口出现前运行启动自检，并根据结果决定是否弹出部署向导。
- 当用户确认环境稳定后，会保存当前部署签名，避免健康环境重复弹窗。

### `core/config.py`
- 提供默认配置。
- 支持从 `config/default_config.json` 加载覆盖项。
- 统一管理模型目录、字体、停用词、阈值、OCR 参数和离线环境变量。
- 提供 `enable_model_trial_load_check`，用于控制启动阶段是否执行真实模型试加载。

### `core/startup_checks.py`
- 负责部署自检的纯逻辑部分。
- 输出结构化检查结果，而不是直接弹窗或打印。
- 检查配置文件、核心依赖、模型目录、真实模型试加载、文档依赖、OCR 管线、Tesseract 语言包、字体和自定义词典。
- 负责比较前后两次自检结果，并生成部署诊断 Markdown 报告。
- 这层不依赖主窗口，因此易于测试和复用。

### `core/text_cleaner.py`
- 对政务文本做降噪。
- 清除新华社快讯、责任编辑、来源等通稿噪音。
- 保留段落和句子结构，供后续比较算法使用。

### `core/algorithms.py`
- 负责单篇分析和双篇对比。
- 单篇分析输出元信息、新提法、核心议题和摘要概览。
- 双篇对比输出措辞微调、动态删减监测、议题新增/保留/强化结果和结构化摘要页数据。
- 仅在需要时懒加载本地 `SentenceTransformer` 模型。

### `core/nlp_thread.py`
- 将 UI 线程与多进程分析隔离开。
- 使用 `QThread` 监听后台任务状态。
- 使用 `ProcessPoolExecutor(max_workers=1)` 保守调度重模型任务。
- 子进程启动时统一设置离线环境变量与 `torch.set_num_threads(2)`。

### `core/result_formatter.py`
- 将分析结果渲染为 Markdown。
- 将分析结果渲染为正式 HTML 报告。
- 将结果序列化为 JSON。
- 输出结构已从“结果清单”升级为“封面页 + 执行摘要 + 证据页”的报告型模板。

### `importers/document_loader.py`
- 负责统一处理 TXT / DOCX / PDF 导入。
- 文本文件支持多编码回退。
- DOCX 基于 `python-docx` 读取段落和表格。
- PDF 优先基于 `pypdf` 提取文字层内容。
- 当 PDF 没有文字层且启用了 OCR 时，会尝试用 `PyMuPDF + Pillow + pytesseract` 做扫描版回退识别。

### `ui/startup_wizard.py`
- 负责把部署检查结果可视化成一个首次部署向导。
- 支持重新检测、打开模型目录、打开配置目录和导出部署诊断报告。
- 会高亮展示修复前后差异，帮助部署人员确认问题是否真的被修复。
- 对关键问题会在进入软件前再次确认。

### `ui/main_window.py`
- 负责输入、按钮、进度、状态与结果展示。
- 提供文档导入、任务启动、取消任务和多格式结果导出。
- 在正常使用过程中也可以重新打开部署自检向导。
- 所有 NLP 调用都只能通过线程桥接层发起。

## 3. 为什么要把部署检查单独拆成 `startup_checks.py`

如果把这些检查直接塞进 `main.py` 或窗口类，会很快出现两个问题：

- 启动逻辑和业务界面逻辑耦合，后续很难测试。
- 同一套检查结果无法在首次向导、手动自检、差异比对和诊断报告导出中复用。

拆出以后，部署检查就成了和分析算法一样的“服务层”。UI 只负责展示，核心判断都集中在同一处。

## 4. 为什么要有“首次部署向导”

很多离线桌面项目并不是“代码不能跑”，而是“环境没配好”。常见问题包括：

- 模型目录没放对
- tokenizer 或权重缺文件
- 模型文件在，但真实加载会报错
- OCR 依赖装了但 Tesseract 不在 PATH
- Tesseract 在，但语言包和配置不匹配
- 字体和自定义词典漏打包

如果这些问题只在用户第一次点击“开始分析”时才暴露，体验会非常差。首次部署向导的意义，就是在真正进入业务功能前，把这些问题提前暴露并给出修复路径。

## 5. 当前实现的边界

这套版本已经具备更像正式产品的部署体验，但仍有边界：

- 部署向导当前只负责检查和引导，不会自动下载模型或安装依赖。
- OCR 依赖本地安装 `Tesseract-OCR`，当前项目不会自动下载安装。
- 真实模型试加载只执行最小化编码验证，还不是完整性能热身。
- 当前测试仍以 smoke test 为主，没有覆盖完整 GUI 交互链路。

## 6. 推荐的下一阶段演进

- 增加模型热身和启动性能基准测试。
- 增加 `tessdata` 目录和语言包路径的更细粒度诊断。
- 增加部署诊断报告的 HTML / PDF 版本。
- 增加 OCR 结果缓存和批量导入模式。
- 增加完整 GUI 自动化测试，进一步提高桌面交付可靠性。
